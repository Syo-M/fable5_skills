# EVALUATION — 品質評価の手法・結果・限界

このリポジトリの品質主張の一次資料。README には要約のみを置き、根拠・手法・限界はすべて本書に集約する。

## 重要な前提(最初に読むこと)

- **本書のスコアは自己評価です。** 採点ペルソナ・ルーブリック・修正サイクルはすべて本リポジトリ
  自身が定義したものであり、「独立したAIペルソナ」は組織的・第三者的な独立性を意味しません。
- AIによる採点は**人間のセキュリティレビューの代替ではありません**(`SECURITY.md` / `governance`)。
- 参考として**第三者AI(ChatGPT)による独立評価は 89/100**(2026-07-02、v2.0.0 に対して。
  「P0修正で92〜93点相当」の付記つき)。その P0 指摘は v2.1.0 で採用・修正済みです。
  自己評価(91.1)との差は、自己評価バイアスの現実的な目安として併記します。

## 1. 3ペルソナ採点

### 手法

- 採点者: 3つのAIペルソナ(Staff Frontend Engineer / Security & Platform Lead / LLMプロンプト設計者)を
  それぞれ独立したエージェントとして起動
- 較正: **90点 = 「無修正で企業がそのまま採用できる」**
- 採点者はドキュメントを読むだけでなく**成果物を実行検証**する(フックへの実ペイロード投入、
  インストーラの実行、レポート数値の再導出、frontmatter パース、相互参照の照合)
- ルーブリック(10次元・加重、計100): トークン効率 12 / 技術的正確性 13 / セキュリティ網羅 14 /
  内部整合性 11 / 発動信頼性 9 / ドメイン網羅 9 / 実行可能性 10 / 強制可能性 8 / ガバナンス 8 / 保守性 6

### v2.0.0 最終結果(2026-07-02、モデル: Claude 系)

| ペルソナ | v1.9.0 | v2.0.0候補 | v2.0.0最終 |
|---|---|---|---|
| Staff Frontend Engineer | 88.1 | 88.6 | **90.4** |
| Security & Platform Lead | 89.4 | 89.7 | **91.5** |
| LLMプロンプト設計者 | 89.3 | 89.6 | **91.3** |
| **平均** | 88.9 | 89.3 | **91.1** |

「候補」→「最終」の差は、3採点者が指名した反証実験(`--max-turns 6` 感度実験)の実施と収束指摘の
即時修正を、**同一ペルソナが差分を実行検証して再レート**した結果(再交渉ではない)。

**推移(平均)**: v1.1.0 84.5 → v1.2.0 86.9 → v1.3.0 88.8 → v1.5.0 87.7 → v1.6.0候補 88.2 →
v1.9.0 88.9 → **v2.0.0 91.1**

**v2.0.0 を MAJOR とした根拠**: `MAINTENANCE.md` の MAJOR 前ゲート(`--runs 3` 発動評価+ペルソナ
採点)を両方満たした計測マイルストーン(破壊的変更なし)。v1.9.1 時点ではゲート未実施のため
3者一致で延期していた。

### 各ラウンドで修正された収束指摘(抜粋)

- v1.6.0 前: フックの「機械強制」過大表現 / `server-boundaries` の App Router グロブ漏れ /
  正規表現漏れ(`oauth` 等)とパス正規化バグ / センシティブパスリストの4箇所漂流 /
  CHANGELOG 監査証跡の破損 / エージェントメモリのデータガバナンス欠如
- v1.9.1: `oauth2`/`authentication` の二層すり抜け / eval 統計の過大表現 / 実行妥当性ガード欠如
- v2.0.0 前: `curl -o`/`wget -O` 等のダウンロード書込みすり抜け / `signup`/`jwt`/`credentials` /
  レポート来歴刻印

## 2. 発動実測(トリガー評価)

「ルールが書いてある」と「発動する」は別の主張。後者を `eval/` ハーネスで**フック観測**により実測
(モデルの自己申告ではない)。手法の詳細と再現手順は `eval/README.md`。

### v2.0.0 正式測定(29本 × 3ラン = 87ラン、全て有効)

- **フルPASS(3/3): 23/29(79%)/ ラン単位発動率: 74/87 = 85%**(3ターン測定)
- 参考比較: v1.8.0 ベースラインは26本×1ランで 12/26 = 46%(同一26本対比 20/26 = 77%)。
  改善の主因は CLAUDE.md の load-first 指示
- **オーバートリガー負例3本すべて 3/3 クリーン**(過剰ロードの副作用は実測ゼロ)
- エージェント単独期待の5本すべて 3/3。レイヤー別信頼性:
  **エージェント > パスルール > 手順スキル > 領域ルールブックスキル**
- 3ターンで 0/3 だった3本は **`--max-turns 6` の感度実験で全て 3/3**
  (測定地平のアーティファクトと実証。`eval/reports/v2.0.0-horizon-experiment.md`)。
  3ターン値(23/29・85%)を保守的な見出し値として維持

### 生データ

## 4. 再現手順

```bash
# 構造検証(数秒)
node .claude/hooks/sensitive-paths.test.mjs && node .claude/hooks/sensitive-bash.test.mjs
node scripts/check-changelog.mjs && node scripts/check-crossrefs.mjs && node scripts/check-sync.mjs
python3 scripts/check_frontmatter.py && node scripts/build-plugin.mjs --check

# 発動実測(リリースグレード、実モデル呼び出し ≈ 90ラン、要 claude CLI)
node eval/run-eval.mjs --runs 3
```

起動し、加重合計を算出する(採点プロンプトの要旨は本書 §1)。
